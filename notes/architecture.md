# Архитектурная записка

## Назначение решения

`homelio` — локальный CLI-инструмент для детерминированного заполнения трех тендерных DOCX-документов по тестовым JSON-данным. Цель решения — воспроизводимо генерировать итоговые файлы без ручной постобработки и без выдумывания значений, которых нет во входных источниках. Для этого логика заполнения вынесена из кода в явный машиночитаемый маппинг, а поведение на пропусках сделано прозрачным: поле либо остается пустым, либо получает заранее заданный маркер.

Такой подход выбран осознанно. В шаблонах уже есть стабильные плейсхолдеры в квадратных скобках, поэтому для базового контура не нужен OCR, эвристический парсинг или генеративная логика. Главный приоритет — надежность, локальный запуск и возможность быстро проверить, из какого источника взялось каждое заполненное значение.

## Основные компоненты

### 1. CLI-слой

Точка входа находится в `src/homelio/cli.py`. CLI реализован на `typer` и содержит две команды:

- `inspect-placeholders` — служебный инструмент для инвентаризации плейсхолдеров в DOCX-шаблонах;
- `generate` — основной сценарий генерации итоговых документов.

CLI не содержит бизнес-логики заполнения документов. Его задача — принять параметры запуска, провалидировать пути, собрать опции генерации и передать их в генератор. Такое разделение упрощает тестирование: отдельно проверяется поведение командной строки, отдельно — сама логика рендера.

### 2. Конфигурация маппинга

Правила заполнения лежат в `config/mappings/documents.yaml`. Для каждого из трех документов в конфиге описаны:

- исходный шаблон;
- имя выходного файла;
- словарь плейсхолдеров;
- источник данных для каждого плейсхолдера;
- форматтер, если нужен специальный вывод;
- повторяющиеся табличные блоки для строк с позициями.

Это ключевое архитектурное решение проекта. Добавление или изменение правил заполнения в первую очередь делается в YAML, а не в коде Python. Благодаря этому генератор остается универсальным и небольшим, а аудит логики упрощается: можно быстро увидеть, какие поля берутся из `profile`, `tender`, `calc` или `system`.

### 3. Ядро генерации

Основная логика находится в `src/homelio/generator.py`. Генератор отвечает за четыре шага:

1. загрузка JSON-источников и YAML-конфига;
2. построение единого контекста выполнения;
3. рендер одиночных плейсхолдеров и повторяющихся строк;
4. запись нового DOCX в `output/`.

Контекст включает четыре группы данных:

- `profile` — реквизиты компании и подписанта;
- `tender` — параметры закупки и позиции из тендерных данных;
- `calc` — расчетные суммы и цены;
- `system` — служебные поля, которые приходят не из JSON, а из параметров запуска.

Отдельно вычисляется секция `derived`. Сейчас в ней есть `price_rows` — объединенный набор строк для ценового предложения. Он собирается из `tender.items` и `calc.items` по `line_no`, чтобы в одном табличном блоке сочетать исходное наименование, НМЦ, цену предложения и итоговые суммы.

### 4. Работа с DOCX

Вместо редактирования документа через высокоуровневый Word API генератор работает с OpenXML-структурой DOCX как с ZIP-архивом. Файл `word/document.xml` читается напрямую, разбирается через `lxml`, затем в XML заменяются текстовые токены и при необходимости клонируются строки таблиц.

У этого подхода две причины:

- он дает точечный и предсказуемый контроль над заменой плейсхолдеров;
- для текущего объема задачи он проще, чем попытка собирать документ заново средствами `python-docx`.

Ограничение этого решения тоже понятно: оно опирается на то, что плейсхолдеры представлены текстом и не “разорваны” сложным форматированием на несколько независимых фрагментов. Для текущих шаблонов это допущение подтверждено reverse engineering-анализом.

## Сценарий генерации

Пользователь запускает:

```bash
uv run homelio generate --document-date 2026-03-26 --outgoing-number "№2603/1"
```

Дальше система работает так:

1. читает тестовые JSON из `materials/Материалы/04. Пример данных JSON`;
2. загружает маппинг из `config/mappings/documents.yaml`;
3. формирует системные поля, включая человекочитаемую дату и исходящий номер;
4. строит производные строки для табличной части ценового предложения;
5. для каждого документа открывает DOCX-шаблон, заменяет плейсхолдеры и сохраняет результат в `output/`.

Поле `Исх. номер заявки` сделано явным внешним вводом через `--outgoing-number`. Это важное продуктовое решение: тестовые JSON такого поля не содержат, поэтому генератор не должен “догадываться” о значении. Если параметр не передан, поле остается пустым.

## Надежность и проверяемость

Минимальный контур качества обеспечивается тремя уровнями:

- `pytest` проверяет извлечение плейсхолдеров, создание всех трех файлов и замену ключевых значений;
- `ruff` страхует от базовых ошибок импорта, синтаксиса и стиля;
- smoke-проверка итоговой команды генерации подтверждает, что репозиторий воспроизводимо выдает три готовых DOCX.

Дополнительно в `notes/reverse_engineering.md` и `notes/field_inventory.yaml` сохранен анализ шаблонов и источников полей. Это снижает риск “скрытой магии”: архитектурные решения опираются на зафиксированные наблюдения, а не на неявные предположения.

## Границы решения и пути масштабирования

Текущая версия закрывает ровно три документа и один детерминированный тестовый набор входных данных. Она не решает задачи извлечения данных из внешних систем, не генерирует новые бизнес-правила автоматически и не пытается интерпретировать неоднозначные поля через LLM.

Если решение развивать дальше, логичный путь такой:

- расширять YAML-маппинг на новые документы;
- добавлять новые форматтеры и типы повторяющихся блоков;
- при необходимости вводить отдельный слой валидации входных JSON;
- только после этого рассматривать AI как вспомогательный модуль для извлечения исходных данных из неструктурированных источников.

Итоговая архитектура остается намеренно простой: маленький CLI, явный конфиг правил, детерминированный генератор и воспроизводимый выходной пакет. Для тестового задания это сильнее и надежнее, чем более “умная”, но менее проверяемая система.
